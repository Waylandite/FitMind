# 对抗性样本设计模式

> 专门构造的挑战性输入，用于测试 Agent 在非理想条件下的鲁棒性。

---

## 1. 设计目标

对抗样本不是要"打败"Agent，而是暴露系统在处理以下情况时的薄弱环节：

1. 跨领域关键词混淆
2. 否定句和自纠错
3. 模糊指代和上下文依赖
4. 中英混合和单位转换
5. 异常输入和边界值

---

## 2. 模式分类

### 2.1 跨领域关键词混淆

**描述**：消息中包含某个意图的关键词，但实际意图属于另一个领域。

| 示例 | 表面关键词 | 实际意图 |
|------|----------|---------|
| `睡觉前吃了蛋白粉` | `睡`（状态） | nutrition（饮食） |
| `练完了喝了蛋白粉` | `喝`（饮食） | workout（训练） |
| `吃了早饭然后去跑步` | `吃`（饮食） | workout（训练） |
| `练胸的时候感觉睡得不够` | `练`（训练） | body_status（状态） |
| `今天记录了睡眠和饮食` | — | query（元讨论） |

**期望行为**：Agent 应以核心事实为主判断意图，不被边缘关键词误导。

---

### 2.2 否定句与自纠错

**描述**：消息中包含对前面内容的否定或自我纠正。

**类型 A：直接否定**

```
"没做卧推，做了哑铃推举"
```
- 不应提取"卧推"作为完成动作
- 应提取"哑铃推举"

```
"今天不是练胸，是练背"
```
- 纠正训练主题
- 如果此前已记录胸训，应触发纠错流程

**类型 B：中途自纠**

```
"卧推60kg，不对其实是55kg"
```
- "60kg"不应作为有效值
- "55kg"是有效值

```
"深蹲80...不...其实是70kg 3x5"
```
- 以最后的有效陈述为准

**类型 C：先肯定后否定**

```
"今天练了腿，不对没练，休息了一天"
```
- 最终状态：没练/休息日
- 不应记录任何训练事实

---

### 2.3 模糊指代

**描述**：消息中包含需要上下文解析的模糊指代。

```
"和昨天一样"
"还是老样子"
"跟上次一样"
"像上周三那样的"
"那个动作的重量改一下"
"刚才那些都行"
```

**期望行为**：
- 如果可以解析（有历史记录）：恢复上下文进行处理
- 如果无法解析：返回澄清请求

---

### 2.4 中英混合与单位转换

**描述**：消息中混合中英文，或使用非标准单位。

**中英混合**：

```
"今天做了bench press 185lbs 5x5, incline db press 55lbs 4x10,
 cable fly 3x15, cardio 20min"
```

- 动作名识别：`bench press` → `卧推`
- 单位转换：185 lbs × 0.4536 = 83.92 kg ≈ 84 kg
- 动作名识别：`incline db press` → `上斜哑铃推举`
- 动作名识别：`cable fly` → `绳索夹胸`
- cardio 识别

**非标准单位**：

```
"深蹲一边20kg的片，杆20，总共60"
"卧推用的是25磅的哑铃"
"跑步5英里"
"游了1500码"
```

**期望行为**：
- 正确识别并归一化为标准单位（kg, km）
- 计算总重量时考虑杠铃杆重量

---

### 2.5 反问与修辞

**描述**：用户使用反问句或修辞表达。

```
"你觉得我今天练了吗？"          → query（不是workout）
"今天还有什么没记录的？"        → query
"这训练量是不是太少了？"        → query（征求建议）
"今天的饮食是不是碳水太多了？"   → query
```

**期望行为**：
- 反问句归类为 query，不提取事实
- 如果确实包含可以提取的事实，应触发确认

---

### 2.6 极度口语化/网络用语

```
"干就完了"
"今天胸怼得爽"
"腿已废"
"练爆了"
"饮食崩了"
"今天废了"
"状态拉满"
"今天摸鱼了"
```

**期望行为**：
- 尝试提取可理解的语义
- "腿已废" → 腿部训练，高强度
- "饮食崩了" → 饮食质量差，记录 rough estimate
- "摸鱼了" → 休息日 / 未训练

---

### 2.7 隐式纠错

**描述**：用户没有明确说"改成"，但话中隐含了对前序的修改。

```
"哦我记错了，其实是周三练的背"
"刚才那个日期不对"
"等一下，重量不是60..."
```

**期望行为**：
- 识别隐含的纠错意图
- 如果上下文足够，执行纠正
- 如果上下文不足，追问

---

### 2.8 双重否定

```
"不是没有练，是练了但没记录"
"不是不想吃，是忘了吃早饭"
"不是没恢复，是还没完全恢复"
```

**期望行为**：
- 正确解析双重否定的含义
- "不是没有练，是练了但没记录" → 有训练，但记录已丢失 / 需要补充记录

---

### 2.9 情绪化/矛盾输入

```
"今天状态爆好，但是练得很烂"
"饮食超级干净，然而体重涨了"
"练得很爽但没感觉有进步"
```

**期望行为**：
- 各方观点均应捕获
- 将矛盾的事实分别记录（如 mood=很好, energy_level=低）

---

### 2.10 元对话

**描述**：用户在与系统对话时讨论对话本身，而非健身内容。

```
"你能记住我说的吗？"
"上次你给我的建议是什么来着"
"你觉得我练太多了吗"
"你有什么要问我的吗"
```

**期望行为**：归类为 query，不提取事实。

---

## 3. 测试用例生成

对于每种模式，建议至少生成 3-5 条测试用例，覆盖：
- 单模式简单场景
- 单模式复杂场景
- 混合多种模式的复杂场景

例如，混合模式：

```
"今天bench了185lbs 5x5，不对应该是175，状态还行，
睡了大概7个半，对了昨天吃了火锅你们记录了吗，没记录的话记一下"
```

这条消息同时包含：
- 中英混合 + 英制单位（lbs → kg）
- 自纠错（185 → 175）
- 多意图（workout + body_status + nutrition + query）
- 涉及历史记录（昨天）
- 元对话（你们记录了吗）
